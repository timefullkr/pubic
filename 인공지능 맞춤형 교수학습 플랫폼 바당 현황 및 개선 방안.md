## 1. 개요

- **정의**: 인공지능 맞춤형 교수학습 플랫폼(AI Education Platform, **AIEP**) — 한 번의 로그인(SSO)으로 민간 에듀테크·교육청 서비스를 활용하고, 블렌디드 수업 데이터를 수집·분석하여 맞춤형 교육 실현을 목표로 하는 플랫폼. 제주 브랜드명 **바당(BADANG)**(Beyond AI-Driven Adaptive Network for Growth).
- **추진 구조**: **11개 시·도 교육청 공동개발**, 위탁기관 **한국지능정보사회진흥원(NIA)**.
- **설계 특징**: 교사 중심 블렌디드 수업 지원, 통합인증(SSO, SAML 2.0), 학습분석 대시보드, **OneRoster·LTI·xAPI 등 국제 표준 기반의 개방형 연계** 구조.

## 2. 추진 경과 (요약)

- '23.3.23. 시도교육감협의회 공동 개발 합의 → 11개 시·도 참여 확정('23.8.), NIA 위탁('23.10.)
- '25.12.22. **플랫폼 오픈** → '26. 1학기 연구·선도학교 시범 운영 → **'26. 2학기 전체 학교 전면 활용**
- '26.5.26. **1차 구축 사업 완료**(잔여 개선과제는 '26.7.까지) / **2차 구축 사업(안): '26.8.~'27.2.**

## 3. 현황

- **1차 구축 범위**: 회원가입·학생 계정, 클래스·수업·학생 배정, 학습현황/학습분석 대시보드, 빅테크(구글·네이버) 연동 — 즉 **LMS 기반 기능** 중심.
- **콘텐츠·기록**: 콘텐츠 약 9만 개(기본 탑재 동영상 약 4만 개 포함), 학생활동 기록→누가기록 활용, NEIS 기반 환경 자동 구성.
- **연계 에듀테크 13종**(SSO 접속): 리딩앤·마타수학·자작자작·레서·러니·엘리스LXP(교과), 미리캔버스·캔바(저작), 띵커벨·패들렛(수업지원), **AI헬피챗(교육용 생성형 AI)**, 스쿨런, 카피킬러.
- **사업비**: AIEP 59,871백만원(지방비, 제주 2,424백만원). ※ 교육부(KERIS) 「AI·디지털 교수·학습·분석 체계」가 별도 트랙으로 추진 중 — 48,100백만원(특별교부금, 제주 배분 2,827백만원).

### 현황 진단 총괄

- **(기반 구축)** 1차 구축으로 계정·수업·기록을 다루는 **공통 기반(LMS)과 국제 표준 연계 구조는 구축 완료** — 콘텐츠 약 9만 개, 에듀테크 13종 연계, NEIS 기반 환경 자동 구성.
- **(핵심 미완)** 다만 AI 교육 서비스의 3층 구조 — **기반(LMS) · AI 추론(생성형 AI) · 안전·데이터 거버넌스** — 중 바당은 **기반 계층까지만 구축된 상태**. 표준 연계 채널도 실제 가동은 인증(SSO)까지로, 도구 결합(LTI)·기록 회송(xAPI)이라는 개방형 설계의 본령은 아직 쓰이지 않고 있음 → **'인공지능 맞춤형'이라는 명칭과 실제 구축 범위 사이의 간극**.
- **(보호 공백)** 그 사이 학생들의 생성형 AI 사용은 이미 일상화되어, 미성년 학생이 **연령 필터·위기감지·기록 등 공적 보호가 미치지 않는 벤더 개별 서비스에서 각자 AI를 쓰는 상태** — 기능 편의의 문제가 아니라 **미성년 학생의 AI 사용에 대한 공적 보호 책무가 이행되지 못하고 있는 문제**.
- **(과제의 성격)** 따라서 남은 과제는 기존 기능의 운영 개선이 아니라 **AI·학습데이터·안전 층위의 신규 구축이라는 투자 성격의 과제**이며, 바당의 재구축이 아니라 **이미 완비된 표준 기반을 활용해 그 위를 채우는 과제**.

## 4. 개선 방안

### 4-1. 현안 문서('26.6.)가 명시한 한계 — AIEP 특이사항

| # | 한계 | 내용 |
| --- | --- | --- |
| ① | **연계 에듀테크 학습데이터 수집 불가** | 플랫폼 자체·빅테크(구글, 네이버) 학습 데이터 외에는 **연계 에듀테크의 학습데이터를 수집하지 못함**. SSO는 **인증(로그인)만** 표준화할 뿐 데이터 회송과는 별개 채널 — 데이터가 쌓이려면 각 에듀테크가 **xAPI 기록 발행(또는 LTI 성적 회송)** 을 구현해 플랫폼에 보내야 하나, 현행 13종 연계는 인증 단계에 그침 → 맞춤형 학습분석이라는 플랫폼 목표와 간극 |
| ② | **학습데이터 분석·활용 기능 미흡** | 현행 분석·대시보드 기능이 미흡하여 **2차 구축 사업 시 고도화 필요** |
| ③ | **AI 모델 부재** | 빅데이터 기반 AI 분석 모델은 **실증만 완료**된 상태로, 실서비스용 AI 모델은 **2차 구축 사업 시 개발 필요**. '인공지능 맞춤형' 플랫폼이지만 현재는 AI 추천·예측과 **플랫폼 내장 학생용 생성형 AI가 없음**(연계 에듀테크 AI헬피챗은 외부 벤더 서비스 — ⑥ 참조) |
| ④ | **에듀테크 사용권 반복 지출 구조** | 플랫폼 운영비 외에 **연계 에듀테크 13종의 사용권 계약비를 매년 벤더별로 반복 지출**해야 하는 구조. 그럼에도 학습데이터는 벤더에 귀속(①)되어 **지출이 교육청의 데이터·자산으로 남지 않음** |
| ⑤ | **교육부 트랙과의 중복·혼선 우려** | 교육부(KERIS) 체계와 **구축 예산 중복 투입 우려**, 유사 플랫폼 병존에 따른 **학교 현장 혼란 우려**. 제주 관련 재원이 양 트랙에 각각 투입 — **지방비 2,424백만원(직접 부담) + 특별교부금 2,827백만원(국가 재원 배분)**. 재원 출처는 다르나 모두 제주 교육에 쓰이는 공적 재원이 유사 기능에 이중 투입되는 구조 |

### 4-2. 생성형 AI 시대 대응 관점의 기능 공백

> 4-1의 ①(데이터 수집 불가)·③(AI 모델 부재)·④(반복 지출 구조)를 학생·교사·교육청 관점에서 구체화하고(⑥~⑫), 생성형 AI 시대의 접근성·품질 요구(⑬·⑭)를 더한 것이다. 학생들은 이미 일상에서 생성형 AI를 쓰고 있으나, 바당에는 그 접근을 **안전하고 형평하게 보장하고 기록으로 남길** 장치가 아직 없다.

| # | 공백 | 현재 상태 | 개선 방안 |
| --- | --- | --- | --- |
| ⑥ | **학생용 대화형(생성형) AI 부재** | 학생이 수업 안에서 직접 쓸 대화형 AI·AI 튜터가 없음. 연계 에듀테크 중 교육용 생성형 AI(AI헬피챗)가 있으나 **외부 벤더 서비스에 SSO로 접속**하는 형태에 그침. 주요 벤더의 소비자용 앱은 **약관상 연령 제한**이 있어 미성년(특히 초등)에게 학교가 공적으로 제공하기 부적합 | 모든 학생이 가정 형편과 무관하게 쓸 수 있는 **공적 대화형 AI 서비스** 제공 — 멀티모달·스트리밍 등 벤더 앱 수준의 품질을 갖추되, **교육청이 학교급·연령별로 정한 모델·기능 정책** 범위 안에서 제공 |
| ⑦ | **AI 수업 도구 부재(교사 통제)** | 교사가 AI의 응답 방식을 수업 목적에 맞게 제어(정답 제시형이 아닌 튜터·소크라테스식 등)하거나, 실시간 수업에서 AI를 활동 도구로 쓸 수단이 없음 | 교사가 **AI의 동작을 수업 단위로 설계·통제**하는 수업용 AI 도구 — 응답 방식 제어(튜터·소크라테스식), 교육과정·교재를 근거로 답하는 **RAG**, 학습 이력 조회·다단계 과제 수행 등 도구 연동(**AI 에이전트, MCP 등 개방형 프로토콜**). 교사 설정은 학생이 우회할 수 없도록 **게이트웨이(서버) 단계에서 강제 적용**. 이를 토대로 수업설계·평가·맞춤 피드백 지원 |
| ⑧ | **AI 송수신 데이터의 확보 불가** | 학생·교사가 외부 AI·에듀테크와 주고받는 **프롬프트·응답 전문을 교육청이 확보하지 못함**(①의 연장). 데이터는 각 벤더에 귀속 | 학생의 모든 AI 사용이 **교육청 관문을 경유**하여 송수신 전문과 **AI의 도구 호출·수행 단계까지 전량 기록·관리(데이터 주권 확보)**. 기록·열람은 **학습지원·안전보호 목적으로 제한**하고, 열람 범위는 **역할·연령별 차등 설계**(예: 학부모는 사용량·주제 요약 중심, 전문 열람은 위기 대응 등 필요 시), **보관기간·파기 기준을 사전에 확정** — 보호와 학생 프라이버시의 양립 |
| ⑨ | **AI 학습활동의 기록 연계 부재** | 바당의 학생활동 기록·누가기록은 **교사 수동 입력** 기반. 학생의 AI 활용 학습활동이 학습기록저장소(LRS)·누가기록으로 이어지지 않음 | AI 학습활동을 **xAPI 표준으로 바당 LRS(학습기록 정본 저장소)에 자동 적재**(학생 식별자는 바당 계정과 동일 신원으로 연계). **교사가 검토·승인한 후** 누가기록·생기부의 **참고 자료**로 활용(자동 기재 아님). 축적된 기록은 학생별 학습 맥락으로 AI에 제공되어 **다음 지도·피드백을 맞춤화**하는 재료가 됨 |
| ⑩ | **미성년 보호 안전장치 부재** | **교육청이 기준을 설정·검증·통제하는** 연령별 유해 응답 필터, 자해·폭력 등 위기 신호 감지·알림 체계가 없음(벤더 자체 필터가 있더라도 그 기준·작동을 교육청이 확인·통제 불가). 공교육 차원의 안전한 대안이 없어 학생이 벤더 앱을 직접 쓰는 동안에는 **공적 보호·기록이 미치지 못함** | 교육청이 직접 기준을 정해 운영하는 **연령별 안전필터·위기감지 체계** — 위기감지는 오탐·미탐을 전제로 **교사의 관찰을 보조하는 수단**으로 설계하고, 감지 신호는 확인 절차를 거쳐 **담임·전문상담(Wee) 체계 및 법정 신고의무 절차와 연계**. 학생의 필터 우회(탈옥), 자료에 심긴 지시로 AI를 조종하는 **프롬프트 인젝션** 등 AI 고유 보안 위협에는 **입력·출력 양방향 검사**와 도구 호출 권한 최소화로 대응 |
| ⑪ | **개인정보 전송의 기술적 통제 부재** | 학생 개인정보의 외부(민간 벤더) 전송이 **학교운영위원회(학운위) 심의·개인정보 제공 동의 등 행정·동의 절차에 의존** — 전송 시점에 실제로 무엇이 나가는지 확인·차단하는 **기술적 통제 수단이 없음** | 외부 AI로 전송 전 **PII 자동 마스킹·가명처리**(자동 탐지의 한계를 전제로 최소 전송 원칙 병행), 벤더에는 가명 식별자만 전달. 아울러 벤더와 **입력 데이터 학습 미활용·보존기간 제한 계약**을 맺어 기술·계약 이중 통제. 미성년 학생의 대화 전문 원본은 국내·교육청 보관 |
| ⑫ | **AI 사용 비용·한도 관리 부재** | 에듀테크별 **개별 계약(사용권) 구조**로 예산이 지속 소요(④)되고, 사용자·학급 단위 사용량 관리 개념이 없음 | 사용자·학급·기간별 **사용량(토큰) 한도**와, 과제 난이도·연령 정책·비용을 함께 고려한 **모델 라우팅**(한도 근접 시 경량 모델 전환 등 수업 연속성 보장 포함) — 예산의 예측 가능성 확보 |
| ⑬ | **접근성·형평성 지원 미흡** | TTS 등 학습 보조가 일부에 그침. 다문화·특수교육 학생, 저사양 기기 환경에 대한 AI 기반 지원 부족 | **TTS·STT·다국어 번역·저사양 대응** 등 AI 기반 접근성 지원으로 모든 학생의 실질적 사용 보장 |
| ⑭ | **AI 응답 품질 관리 부재** | AI가 틀린 내용을 그럴듯하게 답하는 **오답·환각**을 점검·관리하는 체계가 없음 | 교육과정 기반 **평가 문항 세트로 모델 도입·교체 시 품질 검증**, 운영 중 응답 표본 상시 점검. 교육과정·교재 근거 답변(⑦의 RAG)으로 환각 축소 |

### 4-3. 구조적 제약

- **공동개발 구조의 의사결정 속도**: 기능 변경·추가에 11개 시·도와 NIA·사업단 간 합의가 선행되어야 하는 구조로, 개별 교육청 요구의 신속한 반영에 한계. **빠르게 변화하는 AI 기술(신규 모델·서비스)에 대한 기민한 대응이 구조적으로 어려움.**
- **2차 구축의 범위·시차**: AI 모델 개발·분석 고도화는 2차 구축('26.8.~'27.2.) 과제로 이연 — 그 사이에도 학생들의 생성형 AI 사용은 계속되나 공적 보호·기록 장치는 공백.

### 4-4. 종합 — 대응 방향

바당은 계정·클래스·수업·기록·표준 연계(SSO·OneRoster·LTI·xAPI)를 갖춘 **LMS 기반으로서는 안착 단계**이나, '인공지능 맞춤형'의 핵심인 **AI 모델·전체 학습데이터·분석**은 2차 구축 과제로 남아 있다. 나아가 생성형 AI 시대에 요구되는 **학생 접근 보장·데이터 관리·안전장치**(⑥~⑭)는 **현재 확인 가능한 양 트랙(AIEP/NIA, 교육부/KERIS)의 계획 문서 기준으로는 계획이 확인되지 않으며**, 특히 ⑧(송수신 데이터의 교육청 기록·관리)·⑩(교육청 직접 운영 안전장치)·⑫(통합 비용 관리)는 공동개발·벤더 연계라는 두 트랙의 구조상 개별 교육청 단위로 계획되기 어려운 성격의 과제다.

- **공동 플랫폼 트랙으로 해결할 것**: 2차 구축에서 ①~③(데이터 수집 확대, 분석 고도화, AI 모델 개발)이 반영되도록 제주 요구사항을 협의체에 제기. 교육부 체계와 역할 분담을 명확히 하여 중복 투자·혼선(⑤) 예방.
- **제주가 직접 채워야 할 것**: 공동개발의 합의 구조와 2차 구축 일정으로는 채우기 어려운 ⑥~⑭ — **학생용 생성형 AI 접근, 송수신 데이터의 교육청 기록·관리(데이터 주권), 미성년 보호 안전장치, 비용·품질·접근성 관리** — 는 바당을 재구축하는 방식이 아니라, 바당이 이미 갖춘 표준 연계 위에 **제주가 직접 얹는 AI 접근·추론 계층**으로 보완. 결합 경로: 계정·명부는 바당의 **SSO·OneRoster**를 그대로 공유하고, AI 계층은 바당 수업 화면에서 **LTI 외부 도구**로 실행되며(사용자·클래스 맥락이 표준으로 전달), AI 학습활동 기록은 **xAPI**로 바당 LRS에 회송·적재 — 바당의 표준 콘센트에 플러그를 꽂는 방식. 새 모델·기능은 코어 수정 없이 추가되는 구조로 설계하여 AI 변화에 기민하게 대응.

| **NEIS** | 교육행정정보시스템(나이스) — 학적·학사 정보의 원천 시스템 |
| **LMS** | Learning Management System — 학습관리시스템. 수업 개설·운영, 학생 관리, 학습 이력 관리를 담당 |
| **SSO** | Single Sign-On — 통합인증. 한 번의 로그인으로 연계된 여러 서비스를 이용(바당은 SAML 2.0 표준 사용) |
| **OneRoster** | 학급 명부·수강 정보(학생·교사·클래스)와 성적을 시스템 간 주고받는 국제 표준(1EdTech 제정) |
| **LTI** | Learning Tools Interoperability — 외부 학습 도구를 플랫폼 안에 끼워 넣어 실행하는 국제 표준(1EdTech 제정). 실행 시 사용자·클래스 정보가 안전하게 함께 전달됨 |
| **xAPI** | Experience API — "누가 무엇을 학습했다"는 학습활동 기록을 주고받는 국제 표준(미국 ADL 개발) |
| **LRS** | Learning Record Store — xAPI 학습활동 기록을 저장하는 학습기록저장소 |
| **PII** | Personally Identifiable Information — 개인식별정보(이름·연락처 등 개인을 특정할 수 있는 정보) |
| **TTS / STT** | Text-to-Speech(글→음성) / Speech-to-Text(음성→글) 변환 기술 |
| **RAG** | Retrieval-Augmented Generation — 검색증강 생성. AI가 교육과정·교재 등 지정된 자료를 검색해 근거로 삼아 답하는 기법 |
| **MCP** | Model Context Protocol — AI 모델이 외부 도구·데이터에 표준 방식으로 연결되도록 하는 개방형 프로토콜 |
